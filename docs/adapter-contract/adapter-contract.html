Adapter Contract

The contract every adapter must satisfy to be registered in the build engine. Defines interface, determinism rules, required output properties, and test requirements.

Interface

An adapter implements ToolAdapter from @sdlc-agents/core:

// packages/adapters/<tool>/src/index.ts
interface ToolAdapter {
  readonly name: string;                                      // registry key, kebab-case
  render(agents: AgentDef[], ctx: BuildContext): OutputFile[];
  validate?(outputs: OutputFile[]): Diagnostic[];             // optional self-check
}

Rules

R1 Pure & deterministic: same agents + same context → byte-identical output. No timestamps, no random ids, no network calls, no filesystem reads. Determinism is what makes drift detection and snapshot testing possible.
R2 Relative paths: OutputFile.path is relative to the consumer project root and must use / separators.
R3 No filesystem writes: an adapter must not write files itself — the build engine owns all filesystem writes and records each file's hash in .sdlc/build-manifest.json for drift detection.

Required Output Properties

Every generated file must satisfy all three rules:

  1. Carry a generated footer so humans (and drift tooling) know it is machine-owned: the string Generated by sdlc-agents must appear (exact wording may vary per adapter).
  2. Render every load-bearing field of the agent definition: description, inputs (with required markers), workflow (ordered, with ref hints), output_template reference and policies references when present.
  3. Respect model_variants: if the agent declares a variant for this adapter's tool (e.g. model_variants.claude.prompt_append), include it; ignore other tools' variants.

Required Tests

Each adapter package must contain, under src/__tests__/:

Asserts the exact output paths for a representative agent — verifies the adapter targets the right file locations. Asserts agent id, phase, and all workflow steps appear in the rendered output. At least one toMatchSnapshot() on a full rendered file — any format change shows up in review.

Registered Adapters

Adapter name Output Paths Notes
Universal universal AGENTS.md (index) + .sdlc/agents/<id>.md (one per agent) The baseline tier — plain Markdown readable by any AI tool (Codex, Cursor, Windsurf, Gemini CLI, Zed, …). Always keep this target enabled. No auto-routing: users invoke agents by name or by opening the file.
Claude Code claude-code .claude/agents/<id>.md (one per agent) YAML frontmatter: name, description (prefixed [SDLC:<phase>] — drives auto-routing), model (mapped from model_hint: fast → Haiku, balanced → Sonnet, high-reasoning → Opus), tools. Applies model_variants.claude.
Copilot copilot .github/copilot-instructions.md (index) + .github/prompts/<id>.prompt.md (one per agent) Prompt files use mode: agent frontmatter with a one-line description. Applies model_variants.copilot.
Phase 2 (per SA design FR-11) will replace the hardcoded ADAPTERS map in packages/cli/src/commands/build.ts with a plugin interface (adapters: ["@org/sdlc-adapter-x"] in config). Until then, adapters live in this monorepo.

Adding a New Adapter

  1. Create packages/adapters/<tool>/ with src/index.ts exporting a ToolAdapter.
  2. Satisfy all output properties and tests above (path contract + content contract + snapshot).
  3. Register it in the ADAPTERS map in packages/cli/src/commands/build.ts.
  4. Add the target name to sdlc.config.yaml docs and the root README.md.
  5. Add a row to the Registered Adapters table above, including which tool versions were verified and any format caveats.
