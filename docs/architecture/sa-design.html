Agentic SDLC Agents Set

Solution Architecture Document — bộ AI agent definitions tái sử dụng theo hướng Hybrid: Builder core deterministic + Installer UX một lệnh cho toàn bộ vòng đời SDLC.

1. Executive Summary

Agentic SDLC Agents Set là một bộ AI Agent được đặc tả sẵn (agent definitions, prompts, workflows, skills) cho từng giai đoạn của Software Development Life Cycle (SDLC), được đóng gói theo chiến lược Hybrid:

  1. Builder core: duy trì một nguồn chuẩn trong agents/*.yaml, validate bằng schema, và sau đó build ra nhiều định dạng thông qua các adapter.
  2. Installer UX: cung cấp trải nghiệm cài đặt tương tự như npx skills / create-vue: hỏi người dùng họ đang sử dụng tool nào, muốn kích hoạt những agent nào, scope dự án hay user, các biến ngôn ngữ/team, và sau đó tự động sinh cấu hình và các file đầu ra.

Cơ chế runtime vẫn là universal-first: luôn build AGENTS.md + .sdlc/agents/ để mọi AI tool đọc được, cộng thêm native adapter cho tool nào có format riêng mạnh hơn như Claude Code hoặc GitHub Copilot.

Team mới chỉ cần install bộ agents là có ngay quy trình SDLC chuẩn có AI hỗ trợ — không phải tự viết prompt/agent từ đầu. Cùng coding convention, cùng template tài liệu, cùng quy trình review giữa các team. Một nguồn maintain duy nhất → build ra nhiều format cho từng tool. Giống Terraform modules + npx create-* wizard nhưng cho AI agents.

1.1 Định hướng Hybrid so với Agent Skills CLI

Agent Skills CLI như npx skills hoặc gh skill giải tốt bài toán discover/install: chọn skill, chọn agent host, rồi sinh file đúng nơi như .claude/, .cursor/ hoặc .github/. Dự án này không pivot hoàn toàn sang mô hình đó vì bài toán chính rộng hơn một skill đơn lẻ: chuẩn hóa cả agent catalog SDLC có phase, template, policy, adapter contract, generated manifest và test suite.

Giữ agents/*.yaml là source of truth, validate bằng schema, render deterministic qua adapters, có tests và drift detection. Thêm wizard hỏi tool nào, agents nào, scope project hay user, language/team variables, rồi tự tạo config và build lần đầu. Có thể thêm output target dạng SKILL.md để tương thích ecosystem, nhưng vẫn render từ canonical YAML.

2. Problem Statement

2.1 Hiện trạng

  • Mỗi developer/team tự viết prompt, custom instructions và cấu hình agent riêng → chất lượng không đồng đều, trùng lặp công sức.
  • Mỗi AI tool có format cấu hình riêng:
    • Claude Code: CLAUDE.md, .claude/agents/*.md, .claude/skills/, slash commands, hooks, MCP
    • GitHub Copilot: .github/copilot-instructions.md, .github/prompts/*.prompt.md, custom agents
    • Codex CLI: AGENTS.md
    • Cursor: .cursorrules, .cursor/rules/*.mdc
    • Gemini CLI: GEMINI.md
  • Kiến thức nghiệp vụ và quy trình của công ty (coding standard, review checklist, format tài liệu) không được nhúng nhất quán vào AI workflow.
  • Không có cơ chế versioning, testing, hoặc phân phối cho prompt/agent.

2.2 Bài toán cần giải

Làm sao để viết agent một lần, dùng được trên mọi AI tool, phủ đủ các phase SDLC, và phân phối/cập nhật dễ dàng cho nhiều team?

6.1b Trách nhiệm của các lớp trong mô hình Hybrid

  • Schema validation: Kiểm tra tính hợp lệ bằng JSON Schema đối với các định nghĩa agent.
  • Reference resolution: Phân giải các tham chiếu (policies, templates, skills).
  • Multi-layer configuration merging: Hợp nhất cấu hình đa lớp theo thứ tự ưu tiên base → org → team → project → local (5 lớp, xem mục 7.3).
  • Rendering: Kết xuất ra các định dạng đặc thù của từng công cụ thông qua các adapter tương ứng.
  • Prompt linting: Kiểm tra prompt (về độ dài, các từ bị cấm, các placeholder chưa được phân giải).

Phục vụ chủ yếu cho các maintainer thông qua lệnh: pnpm validate, pnpm build, pnpm test, pnpm typecheck, pnpm lint.
  • Host detection: Phát hiện hoặc hỏi người dùng họ đang sử dụng AI host nào: Claude Code, Cursor, Copilot, Codex, Windsurf, Gemini...
  • Agent presets: Cho phép lựa chọn các preset: full SDLC, planning+coding, review+testing, hoặc tùy chỉnh.
  • Scope selection: Cho phép chọn phạm vi lưu file: project-local hoặc user-global (nếu host hỗ trợ).
  • Generate & Build: Tạo file sdlc.config.yaml, chạy validation và thực thi bản build ban đầu.
  • Command isolation: Cô lập và dành riêng các lệnh phát triển cho maintainers, người dùng cuối không cần nhớ các lệnh phức tạp.

Phục vụ người dùng cuối thông qua một lệnh duy nhất: npx sdlc-agents init.

6.2 Adapter Strategy — 2 tầng

Dựa trên chuẩn mở AGENTS.md — chuẩn de-facto được Codex, Cursor, Windsurf, Gemini CLI, Zed, Amp, Roo, Cline, Jules, Devin… cùng hỗ trợ. Chỉ là Markdown thuần + đường dẫn tương đối — mọi AI agent có thể đọc file đều dùng được. Chỉ viết khi tool có cơ chế riêng đáng giá (subagents, slash commands, rule scoping…). Tool mới ra → tự động được hỗ trợ ở Tầng 0, không cần code thêm.

7. Customization & Extensibility Model

"Convention over configuration, but everything is overridable." — zero-config cho người mới, override từng dòng prompt cho người nâng cao, không bao giờ phải fork repo gốc.

CấpTênNgười dùng làm gìEffort
L1ConfigureBật/tắt agents, set variables (stack, language, team) trong config5 phút
L2OverrideThay thế từng phần: policies, templates, checklist, từng step trong workflow30 phút
L3ExtendViết agent mới của riêng dự án, kế thừa (extends) agent có sẵn1–2 giờ
L4Plug-inViết adapter cho tool mới, hook vào build pipeline, publish preset cho team khác dùng1–2 ngày
 
# sdlc-agents.config.yaml
extends: "@org/sdlc-agents-base"        # hoặc preset cộng đồng: "@sdlc-agents/preset-fintech"
targets: [universal, claude-code, copilot]   # universal phủ mọi tool còn lại

variables:
  language: vi
  stack: nestjs-postgres
  team: payment-squad

agents:
  enable: [planner, coder, code-reviewer]
  disable: [ux-reviewer]

Cơ chế deep-merge có độ ưu tiên rõ ràng (tương tự cấu hình ESLint/Tailwind):

Base package (@org/sdlc-agents-base)     ← độ ưu tiên thấp nhất
  ↑ ghi đè bởi
Org preset (quy chuẩn chung của tổ chức)
  ↑ ghi đè bởi
Team preset (quy ước của từng team)
  ↑ ghi đè bởi
Project config (sdlc-agents.config.yaml) ← độ ưu tiên cao nhất
  ↑ ghi đè bởi
Local overrides (sdlc-agents.local.yaml — được gitignore, phục vụ test cá nhân)

Override chỉ chứa phần khác biệt → khi base cải tiến prompt, user hưởng cải tiến mà vẫn giữ tùy chỉnh riêng. Ba toán tử chính được hỗ trợ: replace (thế toàn bộ), append/prepend (thêm vào cuối/đầu), và insert_after/insert_before (chèn vào vị trí cụ thể trong workflow).

# overrides/code-reviewer.yaml
agent: code-reviewer
patch:
  workflow:
    insert_after: "Chạy checklist security"
    step: "Kiểm tra naming theo quy ước payment-squad"
  output_template: ./our-review-template.md
  prompt_append: |
    Luôn kiểm tra idempotency cho mọi API thanh toán.
Tình huống (Scenario)Lối thoát (Escape Route)
DSL không biểu diễn được yêu cầu đặc thù Sử dụng block tool_specific: trong file agent YAML — viết raw content dành riêng cho 1 tool, các tool khác sẽ bỏ qua block này.
Muốn sửa đổi thủ công file được sinh ra (output file) Chế độ eject: chạy npx sdlc-agents eject code-reviewer — copy file generated thành file thường, đánh dấu managed: false để build engine không ghi đè lên nữa.
File generated vô tình bị sửa đổi thủ công Build engine chèn header # generated — do not edit và chạy các câu lệnh diff để cảnh báo về sự sai lệch (drift) giữa file gốc và file đầu ra.
Cần quay lại trạng thái mặc định ban đầu Chạy npx sdlc-agents reset <agent> — xóa các overrides và khôi phục lại trạng thái của base agent ban đầu.

14. Open Questions

Các câu hỏi mở cần giải quyết trước khi chuyển sang các giai đoạn tiếp theo:

Q1 — Ngôn ngữ prompt: Đã chốt

Prompt được viết hoàn toàn bằng tiếng Anh để đảm bảo chất lượng suy luận cao nhất của model; các output template sẽ tuân theo cấu hình language (mặc định là en, có thể cấu hình sang vi).

Q2 — Công cụ ưu tiên thứ hai: Đã chốt

Chọn GitHub Copilot — các native adapters đợt 1 bao gồm Claude Code và GitHub Copilot.

Q3 — OSS vs Private — Khác biệt trong 3 điểm kiến trúc cốt lõi:
Khía cạnh ảnh hưởng Open Source (Mã nguồn mở) Private-only (Nội bộ)
Plugin interface Phải ổn định, đánh phiên bản (SemVer), có tài liệu đầy đủ vì sẽ đóng vai trò public API. Breaking changes yêu cầu tăng major version. Chỉ phục vụ nội bộ, thoải mái thay đổi cấu trúc mã nguồn (refactoring) khi cần.
Policy layer Không thể nhúng các chính sách bảo mật/nghiệp vụ riêng của doanh nghiệp vào repo chính; bắt buộc phải tách biệt thành packages/policies-private. Nhúng trực tiếp vào monorepo, đơn giản và đồng bộ hơn.
Adapter code Đòi hỏi có tài liệu kiểm thử E2E đầy đủ + cam kết tương thích bền vững cho cộng đồng phát triển. Chỉ cần đảm bảo chạy đúng nghiệp vụ nội bộ.

Build engine, DSL schema, và cơ chế vendor sử dụng mã nguồn giống hệt nhau, không có rủi ro phân nhánh kiến trúc.

Có nên thương mại hóa? Đề xuất: Open Core.

  • OSS (miễn phí): core engine + Universal adapter + bộ catalog agents cơ bản → tạo cộng đồng, tăng độ phủ, nhận đóng góp adapter từ cộng đồng.
  • Commercial (trả phí): giao diện hosted policy management (dashboard quản lý team, audit log, vault chứa secret), bộ native adapters đầy đủ, hosted eval harness, hỗ trợ SLA, marketplace preset chuyên ngành.
  • Tiền lệ: Terraform OSS → HCP, ESLint → công cụ doanh nghiệp, Biome → enterprise tier.

Action item: Quyết định trước Phase 1. Nếu chọn OSS, thiết kế giao diện plugin như một public API ngay từ đầu (SemVer + deprecation policies) và cô lập packages/policies-private ra khỏi repository chính.

Q4 — Ai sở hữu bộ quy chuẩn chính sách chung của tổ chức?

Cần một nhóm chịu trách nhiệm (ví dụ: nhóm Engineering Excellence) để kiểm duyệt và phê duyệt các thay đổi đối với checklist/convention chung của doanh nghiệp.

Q5 — Độ sâu tích hợp MCP (Model Context Protocol):

Chúng ta nên đi kèm các MCP server riêng (ví dụ: kết nối Jira, Confluence) ngay trong bộ agents này hay để dành cho các phase sau?

Q6 — Cơ chế kích hoạt ở tầng Universal:

Claude Code hỗ trợ tự động kích hoạt kỹ năng dựa trên mô tả, nhưng đối với các công cụ chỉ đọc file AGENTS.md thì agent được gọi như thế nào? Model sẽ tự đọc file index hay người dùng phải hướng dẫn bằng lệnh thủ công (ví dụ: "hãy tuân theo .sdlc/agents/planner.md")? Cần thực hiện spike thử nghiệm Phase 0 trên 2-3 công cụ thực tế vì nó quyết định trải nghiệm người dùng tầng Universal.

Q7 — Ngân sách chạy Đánh giá (Evaluation):

Eval harness gọi các API model thực tế (Claude/GPT/Gemini) — ai sẽ trả tiền, hạn mức cho mỗi lần chạy CI ra sao? Cần có cơ chế kiểm soát quota để tránh việc CI bỏ qua đánh giá do cạn kiệt tài khoản.

Q8 — Pháp lý khi thương mại hóa:

Yêu cầu ghi nhận bản quyền (attribution) đối với các kỹ năng từ cộng đồng sử dụng giấy phép MIT/Apache khi đóng gói vào sản phẩm thương mại. Đồng thời, tên sản phẩm có xảy ra tranh chấp thương hiệu không? Cần review pháp lý trước khi release chính thức ra công chúng.

15. Định vị vs Hệ sinh thái Skill

Hiện nay có rất nhiều bộ quy tắc và kỹ năng nổi tiếng như obra/superpowers, anthropics/skills, awesome-claude-code, awesome-cursorrules, agent-rules... Sản phẩm này không cạnh tranh ở cấp độ nội dung của từng skill đơn lẻ — nó cạnh tranh ở cấp độ hệ thống:

"Skill ngoài kia là bài hát lẻ; sản phẩm này là hãng phát hành album — có tuyển chọn, chuẩn hoá, đóng gói đa nền tảng, phân phối và cập nhật."

15.2 Tái sử dụng Kỹ năng Sẵn có

Chiến lược: KHÔNG viết lại những gì cộng đồng đã làm tốt. Thay vào đó, chúng ta xây dựng cơ chế import trực tiếp trong build engine:

# agents/bug-triager.yaml
id: bug-triager
imports:
  - source: github:obra/superpowers            # high-star GitHub skill
    path: skills/systematic-debugging
    pin: v4.2.0                                # ghim cụ thể tag/commit — tránh trôi phiên bản
    license: MIT                                # build lỗi nếu license không tương thích
prompt_prepend: |
  Apply the systematic-debugging workflow below, outputting in {{language}}.
  • Vendor + pin: Các skill được tải về và sao chép vào thư mục vendor/ khi build, được ghim cố định bằng commit hash/tag nhằm đảm bảo quá trình build luôn có tính deterministic và hoạt động offline tốt, không phụ thuộc vào trạng thái online của repo nguồn.
  • Tự động kiểm tra bản quyền: Chỉ cho phép import các skill có license tương thích (ví dụ: MIT/Apache-2.0); tự động chèn ghi nhận bản quyền (attribution) vào output đầu ra.
  • Wrap, don't modify: Giữ nguyên vẹn mã nguồn của skill gốc, chỉ tùy biến qua các block prompt_prepend hoặc prompt_append. Khi repo nguồn phát hành phiên bản mới, ta chỉ việc nâng chỉ số pin lên phiên bản mới và chạy lại các evals test.
  • Các ứng viên tiềm năng để import cho MVP: Quy trình gỡ lỗi hệ thống (systematic-debugging) & TDD workflow (từ repo superpowers, ~94k+ stars trên Anthropic Marketplace), các kỹ năng làm tài liệu từ anthropics/skills (~135k stars), các kỹ năng workflow kỹ thuật của mattpocock/skills (~124k stars, giấy phép MIT — xem Bảng 15.3), các agent riêng lẻ từ VoltAgent/awesome-claude-code-subagents và wshobson/agents. Việc tái sử dụng này giúp giảm đáng kể công sức tự viết 6 agent MVP — phần chúng ta phải tự phát triển chủ yếu là các chính sách riêng của công ty + template tiếng Việt + logic liên kết các tác nhân thành một vòng khép kín.
