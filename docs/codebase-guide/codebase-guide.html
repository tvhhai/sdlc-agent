Hướng dẫn thư mục Codebase (Codebase Folder Guide)

Hướng dẫn đầy đủ về cấu trúc repo sdlc-agent theo hướng Hybrid — từng folder, từng package, phần nào là source, phần nào generated, và đánh giá đối chiếu SA design với code thực tế.

0. Dự án này làm gì và dùng như thế nào

sdlc-agent là một monorepo TypeScript để định nghĩa, validate, build và phân phối một bộ AI agents cho các giai đoạn SDLC: requirement, planning, architecture, coding, review và testing. Thay vì copy/paste prompt riêng cho từng tool, project giữ một nguồn chuẩn trong agents/*.yaml, sau đó render ra format phù hợp cho nhiều AI tools.

Định hướng sản phẩm hiện tại là Hybrid:

  • Builder core: schema, validation, adapter render, drift detection và test suite vẫn là nền kỹ thuật chính.
  • Installer UX: user cuối không cần nhớ chuỗi lệnh dài; npx sdlc-agents init sẽ hỏi đang dùng AI host nào, muốn bật agents nào, scope project hay user, các biến ngôn ngữ/đội ngũ, rồi tự tạo cấu hình và chạy build lần đầu.
  • Chuẩn hóa workflow AI agent cho đội ngũ phát triển.
  • Quản lý version, schema, test cho các định nghĩa agent.
  • Build một lần ra nhiều target: Universal (AGENTS.md), Claude Code, và Copilot.
  • Cho phép tùy chỉnh bằng config, policy, template, tránh sửa tay file generated.
 Đã hoàn thành 6 agents cơ bản, CLI tích hợp init / validate / build, 3 adapters, cơ chế validate liên kết và phát hiện drift. Các tính năng nâng cao như merge config nhiều lớp, inheritance (extends), vendoring (imports) và hệ thống test tự động (eval harness) sẽ được triển khai ở Phase 2+.

Cách dùng nhanh cho consumer project

Để khởi tạo nhanh dự án tiêu dùng, hãy chạy lệnh:

npx sdlc-agents init

Wizard sẽ hướng dẫn chọn AI host, agent preset, phạm vi cài đặt và các biến môi trường, sau đó tạo tệp sdlc.config.yaml và tự động kết xuất đầu ra lần đầu. Các lệnh phát triển trong repo này như pnpm sdlc validate, pnpm sdlc build, hay pnpm test dành riêng cho duy trì và đóng gói hệ thống.

1. Tổng quan kiến trúc

Hệ thống được tổ chức thành 4 lớp chính như sơ đồ Solution Architecture:

  1. Canonical layer: Chứa mã nguồn chuẩn của agents (YAML), templates, policies và config.
  2. Build engine: Nhân xử lý nạp config, đọc agents, validate Zod schema, resolve biến và phân phối cho các adapter.
  3. Installer / Wizard UX: Giao diện cài đặt tương tác thân thiện qua CLI (Phase 2).
  4. Tool adapters: Render nội dung chuẩn ra các định dạng đích cụ thể.

Luồng xử lý (Build Flow)

flowchart LR
  A["sdlc.config.yaml"] --> B["packages/cli runBuild"]
  C["agents/*.yaml"] --> D["packages/core loadAgents"]
  D --> E["AgentSchema validation"]
  B --> F["resolveAgent variables"]
  G["templates/*.md"] --> H["BuildContext"]
  I["policies/*.md"] --> H
  F --> J["ToolAdapter.render"]
  H --> J
  J --> K["AGENTS.md and .sdlc/"]
  J --> L[".claude/agents/"]
  J --> M[".github/copilot-instructions.md and .github/prompts/"]

2. Root folder và root files

Các tệp tin cấu hình và tài liệu chính tại thư mục gốc của monorepo:

package.json Khai báo cấu hình monorepo, quy định package manager pnpm@9.15.0. Chứa các scripts điều phối: pnpm sdlc (chạy CLI), pnpm test (vitest), pnpm typecheck (tsc), pnpm lint (biome check). Khai báo các dependencies chung (zod, yaml, tsx).
pnpm-workspace.yaml Định nghĩa các workspace con trong monorepo: packages/* và packages/adapters/*, cho phép liên kết thư viện nội bộ qua cú pháp workspace:*.
pnpm-lock.yaml Khóa phiên bản các package dependencies và liên kết trực tiếp các package nội bộ trong workspace.
tsconfig.json Cấu hình TypeScript dùng chung. Target ES2022, module resolution dạng bundler, bật chế độ kiểm tra nghiêm ngặt (strict: true). Khai báo alias @sdlc-agents/core trỏ về source code thực tế.
vitest.config.ts Cấu hình Vitest runner. Sử dụng pattern bao gồm cả file tests ở adapter: packages/*/src/**/*.test.ts và packages/adapters/*/src/**/*.test.ts để đảm bảo test suite chạy đầy đủ.
sdlc.config.yaml Cấu hình kết xuất của dự án. Khai báo danh sách các adapter mục tiêu cần render (universal, claude-code, copilot) và các biến thay thế toàn cục (language: en).
AGENTS.md Tệp đầu ra tự động được tạo bởi universal adapter. Đóng vai trò chỉ mục và hướng dẫn kích hoạt agent trên các công cụ AI hỗ trợ đọc tài liệu. Không sửa tay file này.
.gitignore Loại trừ các thư mục build, caches, node_modules, log, và đầu ra thử nghiệm của thư mục spike/output/.

4. templates/ — Output Templates

Mẫu định dạng tài liệu mà các AI agent sử dụng khi kết xuất kết quả đầu ra (artifacts):

  • plan.md — Bản mẫu kế hoạch triển khai chi tiết cho planner.
  • review-report.md — Bản mẫu báo cáo đánh giá mã nguồn cho code-reviewer.
  • prd.md — Bản mẫu tài liệu đặc tả yêu cầu sản phẩm cho requirement-analyst.
  • hld.md — Bản mẫu thiết kế kiến trúc cao cấp và các quyết định kỹ thuật (HLD/ADR) cho solution-architect.
  • test-plan.md — Bản mẫu kế hoạch kiểm thử tự động cho test-generator.
Build engine tải các template này và truyền vào BuildContext.templates. Hiện tại, adapters render liên kết tương đối để hướng dẫn AI agent mở tệp template tham chiếu thay vì inline trực tiếp toàn bộ nội dung tệp template vào prompt.

5. policies/ — Checklist và Coding Policy

Các tài liệu quy chuẩn, danh sách kiểm tra (checklists) bắt buộc áp dụng khi các agent thực hiện công việc:

  • conventions.md — Quy tắc đặt tên, hướng dẫn giới hạn độ dài hàm, xử lý lỗi (error handling), kỳ vọng viết test và quy tắc TypeScript nghiêm ngặt.
  • security-checklist.md — Hướng dẫn kiểm tra dữ liệu đầu vào (input validation), phân quyền (auth/authz), quản lý secrets, đánh giá dependencies và chính sách ghi logs/bảo vệ thông tin PII.
Tương tự templates, policies được đưa vào context để adapter render đường dẫn tham chiếu. Universal adapter liên kết trực tiếp bằng link tương đối; Claude adapter liệt kê chuỗi path; Copilot adapter hiện chưa hỗ trợ kết xuất phần này (ngầm áp dụng qua workflow).

6. packages/core/ — Build Engine Core

Thư viện xử lý lõi của build engine (@sdlc-agents/core), được viết hoàn toàn độc lập với giao diện CLI hay bất kỳ adapter cụ thể nào.

package.json Khai báo package ESM nội bộ. Export các API thông qua entrypoint src/index.ts. Đăng ký dependencies cần thiết như zod và yaml.
src/schema.ts Định nghĩa Zod schema và kiểu dữ liệu (TypeScript types) cho cấu trúc của Agent (AgentDef). Bao gồm validation nghiêm ngặt: kiểm tra semver, kebab-case id, độ dài tối thiểu của mô tả, danh sách whitelist các loại license cho việc import skill (MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause).
src/types.ts Chứa các định nghĩa interfaces dùng chung cho toàn dự án: OutputFile (lưu path relative và content text), Diagnostic (cảnh báo lỗi), ResolvedConfig, BuildContext, và ToolAdapter interface.
src/config.ts Đảm nhiệm đọc tệp sdlc.config.yaml từ project root. Trả về cấu hình mặc định (default config) nếu không tìm thấy tệp cấu hình, thực hiện validate Zod schema trước khi trả ra config hoàn chỉnh.
src/loader.ts Quét toàn bộ thư mục agents/ để tìm các tệp .yaml, sắp xếp tên tệp theo thứ tự bảng chữ cái để đảm bảo tính bất biến (deterministic execution), sau đó giải mã YAML và validate schema.
src/merger.ts Hiện thực hóa mergeConfigs(base, ...overrides) để gộp các cấu hình nhiều lớp (đè biến, đè mảng targets) mà không gây biến đổi cấu hình gốc (mutating base).
src/resolver.ts Chứa logic phân tích và thay thế biến động trong văn bản {{variable}}. Thực hiện đệ quy sâu (deep recursive traversal) trên đối tượng agent giúp giữ nguyên kiểu dữ liệu và cấu hình nguyên bản, tránh làm hỏng cấu trúc chuỗi JSON.
src/__tests__/ Thư mục unit tests toàn diện của core. Bao phủ toàn bộ các module: validate schema (các trường hợp fail cấu hình), loader (bỏ qua file lạ), merger (ưu tiên đè biến cấu hình sau), resolver (giữ nguyên khi biến trống, xử lý nhiều biến cùng lúc).

7. packages/cli/ — Command Line Interface

Giao diện dòng lệnh chính thức (@sdlc-agents/cli) điều phối luồng làm việc của build engine.

package.json Khai báo lệnh nhị phân sdlc trỏ vào entrypoint. Chứa khai báo dependencies bao gồm core, 3 adapters và thư viện xử lý lệnh commander.
src/index.ts Khởi tạo giao diện dòng lệnh commander, khai báo các lệnh: sdlc init (khởi tạo cấu hình), sdlc validate (kiểm tra DSL), sdlc build (kết xuất file).
src/commands/build.ts Luồng build chính: load config ➔ load agents ➔ validate toàn diện liên kết chéo ➔ duyệt mảng targets active ➔ render qua các adapter tương ứng ➔ ghi tệp tin ra đĩa.
Drift Detection: Tính toán mã băm SHA-256 của toàn bộ tệp đầu ra và ghi vào .sdlc/build-manifest.json. Khi chạy build lần sau, nếu phát hiện tệp đã bị người dùng chỉnh sửa thủ công, engine sẽ đưa ra cảnh báo "Drift detected" trước khi ghi đè.
src/commands/validate.ts Thực hiện tải các agents và chạy validate liên kết dự án, báo lỗi Zod định dạng chi tiết khi phát hiện file YAML sai quy định.
src/commands/project-validation.ts Chứa hàm validateProjectContracts. Báo lỗi và dừng chương trình khi: trùng lặp agent id, thiếu tệp template tham chiếu, thiếu tệp policy, hoặc agent cố sử dụng các trường nâng cao chưa được hỗ trợ (extends, imports).
src/commands/init.ts Tạo tệp sdlc.config.yaml mặc định (bật cả 3 targets) nếu tệp cấu hình chưa tồn tại.

8. packages/adapters/ — Render Targets

Các package adapters thực hiện chuyển đổi định dạng trung gian thành prompt config cụ thể cho từng IDE agent:

Mục tiêu: Tạo baseline portable hoạt động trên mọi công cụ AI hỗ trợ đọc file trong repo (Codex, Cursor, Windsurf, Gemini, Zed...).

Kết quả đầu ra: Tệp mục lục AGENTS.md ở root và các file Markdown chi tiết của từng agent tại .sdlc/agents/<id>.md.

Đặc điểm: Sắp xếp cấu trúc gọn gàng, hiển thị rõ ràng danh sách tham số đầu vào và quy trình thực hiện kèm theo các liên kết tương đối trỏ về thư mục policies.

Mục tiêu: Tạo subagents tự động định tuyến cấu hình riêng cho Claude Code CLI.

Kết quả đầu ra: Các tệp Markdown tương ứng tại thư mục .claude/agents/<id>.md.

Đặc điểm: Thêm định dạng YAML frontmatter chuẩn (gồm name, description mang mã phase dạng [SDLC:<phase>] để Claude tự định tuyến, model, tools). Tự động ánh xạ model thích hợp dựa trên model_hint (Haiku cho fast, Sonnet cho balanced, Opus cho high-reasoning) và chèn cấu hình tùy biến riêng model_variants.claude.prompt_append.

Mục tiêu: Tạo prompts tùy chỉnh định tuyến cho GitHub Copilot Chat.

Kết quả đầu ra: Cấu hình chung .github/copilot-instructions.md và các tệp prompt chi tiết tại .github/prompts/<id>.prompt.md.

Đặc điểm: Sử dụng định dạng YAML frontmatter với mode: agent và chèn cấu hình prompt bổ sung model_variants.copilot.prompt_append.

Yêu cầu bắt buộc về kiểm thử cho mọi Adapter

  • Path contract test: Bảo đảm tệp tin đầu ra được đặt đúng vị trí thư mục quy định của IDE target.
  • Content contract test: Bảo đảm chứa đủ thông tin định danh và các bước workflow của agent trong prompt.
  • Snapshot test: Dùng toMatchSnapshot() trên toàn bộ nội dung tệp đầu ra nhằm phát hiện lập tức bất kỳ sự lệch định dạng nào khi sửa đổi code adapter.

11. spike/ — Prototype Phase 0

Thư mục chứa tệp tin thử nghiệm spike/render.ts là bản prototype độc lập từ Phase 0:

  • Định nghĩa schema Zod nội bộ độc lập, không sử dụng import từ các package core.
  • Đọc trực tiếp các tệp YAML trong thư mục agents/ và xuất file thử nghiệm ra spike/output/.
  • Giúp kiểm chứng nhanh tính khả thi của mô hình kết xuất prompt trước khi tách mã nguồn thành monorepo chính thức.
Mã nguồn của spike không được sử dụng ở runtime chính. Toàn bộ mã nguồn chính thức nằm tại packages/.

13. skills/ — Kỹ năng bổ sung (Placeholder)

Thư mục skills/ hiện tại đang để trống. Theo thiết kế kiến trúc, đây sẽ là nơi lưu trữ các bộ kỹ năng chi tiết (workflows, scripts) hỗ trợ tích hợp hoặc đóng gói sâu cho các IDE agents dưới dạng local hoặc vendored skills.

Mặc dù Schema đã cho phép khai báo trường imports, build engine hiện tại vẫn chưa triển khai logic kết nạp hay kiểm tra giấy phép cho các skill ngoại vi này (đã lên kế hoạch ở Phase 2).

14. Mối quan hệ giữa policies, templates, và agents

Ba thành phần này tạo nên lớp dữ liệu chuẩn hóa của dự án, hỗ trợ cung cấp bối cảnh và quy chuẩn định dạng cho AI:

flowchart TD
  A["agents/*.yaml"] --> B["output_template: templates/*.md"]
  A --> C["policies: policies/*.md"]
  B --> D["AI agent follows output format"]
  C --> E["AI agent follows checklist/convention"]
Build engine của dự án đã bổ sung cơ chế xác thực liên kết. Nếu một tệp YAML agent tham chiếu đến file template hoặc policy không tồn tại, chương trình build sẽ ngay lập tức báo lỗi và dừng thực thi để tránh tạo ra prompt lỗi.

15. Chiến lược kiểm thử hiện tại

Hệ thống tập trung kiểm tra tính đúng đắn ở mức biên dịch tĩnh (static correctness) và tính bất biến của kết quả đầu ra:

  • Schema Correctness: Bảo đảm các định nghĩa agent YAML viết đúng cú pháp.
  • YAML Loading & Config Merge: Kiểm tra tính chính xác của cơ chế gộp cấu hình.
  • Variable Resolution: Bảo đảm việc giải biến {{variable}} hoạt động đúng mà không làm hỏng cấu trúc.
  • CLI Build & Validate: Thử nghiệm thực thi các lệnh CLI thực tế.
  • Adapter Snapshots: So khớp chính xác nội dung kết xuất để tránh thay đổi ngoài ý muốn khi sửa code adapter.

Lệnh chạy toàn bộ test suite:

pnpm test
pnpm typecheck

16. Quy trình thêm một Agent mới

Để thêm một agent mới vào catalog, hãy thực hiện theo các bước sau:

  1. Tạo file định nghĩa agent mới dưới dạng YAML tại agents/<new-agent-id>.yaml.
  2. Thiết lập các trường bắt buộc (id tuân thủ kebab-case, version dạng semver, chọn đúng phase trong enum, mô tả description ≥ 10 ký tự).
  3. Xác định các tham số inputs, các bước workflow và các công cụ cần thiết ở tools_required.
  4. Nếu cần quy định định dạng đầu ra, tạo tệp Markdown tương ứng trong templates/. Nếu cần quy chuẩn kiểm tra, tạo tệp trong policies/, sau đó tham chiếu tương ứng trong file YAML.
  5. Chạy lệnh kiểm tra tính hợp lệ: pnpm sdlc validate.
  6. Nếu không có lỗi, chạy build để xuất các file cấu hình tương ứng cho các adapter: pnpm sdlc build.
  7. Chạy bộ unit test để đảm bảo không có lỗi TypeScript hoặc kiểm thử snapshot bị lệch: pnpm test và pnpm typecheck.

17. Quy trình thêm một Adapter mới

Quy trình đóng gói kết xuất cho một công cụ AI hoặc IDE agent mới:

  1. Tạo thư mục package mới tại packages/adapters/<tool-name>/.
  2. Tạo tệp package.json và khai báo dependency vào package @sdlc-agents/core.
  3. Hiện thực hóa một class export từ package kế thừa interface ToolAdapter và triển khai method render(agents, ctx) trả ra danh sách OutputFile[].
  4. Viết đầy đủ 3 loại tests (path contract test, content contract test, snapshot test) trong thư mục src/__tests__/ của adapter.
  5. Thêm package adapter mới vào dependencies của package packages/cli/package.json.
  6. Import và đăng ký adapter mới vào đối tượng ADAPTERS tại tệp packages/cli/src/commands/build.ts.
  7. Khai báo thêm tên target mới vào tài liệu cấu hình sdlc.config.yaml để bắt đầu sử dụng.

18. Các điểm cần chú ý khi phát triển tiếp

CodeGraph Dự án hiện chưa khởi tạo CodeGraph. Khi cần phân tích kiến trúc dự án ở mức độ rộng hoặc trả lời câu hỏi cấu trúc lớn, có thể chạy lệnh codegraph init -i và sử dụng các công cụ tương tác.
Encoding Output hiển thị Một số terminal trên hệ điều hành Windows có thể hiển thị sai phông chữ tiếng Việt hoặc Unicode (mojibake). Tập tin gốc vẫn được đảm bảo lưu dưới định dạng UTF-8; hãy kiểm tra bằng các trình biên soạn hỗ trợ UTF-8 khi cần xác minh nội dung.
Template Engine Hệ thống template hiện tại chỉ cung cấp hướng dẫn tĩnh cho AI agent hoạt động. Dự án chưa có trình phân tích cú pháp để render trực tiếp dữ liệu động vào template. Nếu cần tạo ra tài liệu hoàn chỉnh tự động, cần bổ sung một template renderer engine.
Config Multi-layer Mặc dù nhân core đã hỗ trợ hàm mergeConfigs() phục vụ cấu hình nhiều lớp (base -> org -> team -> project -> local), CLI hiện tại mới chỉ kết nối để đọc file sdlc.config.yaml đơn lẻ.
Imports & Extends mới Schema Zod đã chấp nhận các trường imports và extends để chuẩn bị cho Phase 2, nhưng build engine hiện tại chưa triển khai logic kế thừa agent hay import kỹ năng. Validator hiện tại sẽ chặn lỗi cảnh báo rõ ràng khi agent cố dùng các trường này để tránh lỗi không mong muốn.
CI Workflow cần review Tệp tin .github/workflows/ci.yml hiện tại đang cấu hình các đường dẫn (paths) và thư mục làm việc theo thư mục con AI_Agent_SDLC_set/. Do mã nguồn dự án hiện đang được phân bố trực tiếp tại root của repository, cần chỉnh sửa lại workflow để CI chạy đúng phạm vi.

20. Mental Model ngắn gọn

YAML mô tả chi tiết agent, inputs, workflow steps, tools, policies. Phục vụ parse YAML, validate Zod schema, resolve biến môi trường. Đọc config, nạp templates, điều phối nạp data cho các adapter. Chuyển đổi agent chuẩn thành file prompt tương thích với IDE target. Đầu ra tự động, được commit để IDE/AI agents đọc sử dụng ngay. Các file markdown hướng dẫn định dạng và quy tắc bảo mật cho agent.

21. Đánh giá bổ sung sau khi review SA design và codebase 2026-06-11

Phần này ghi nhận các phân tích đối chiếu thực tế giữa thiết kế giải pháp Solution Architecture ban đầu và hiện trạng triển khai mã nguồn hiện tại.

  • Giải quyết trực tiếp pain point thực tế của nhà phát triển: tránh việc sao chép/dán prompt thủ công giữa các công cụ AI khác nhau.
  • Kiến trúc phân lớp canonical DSL ➔ build engine ➔ adapters rất rõ ràng và dễ bảo trì.
  • Định hướng Universal-first giúp hỗ trợ ngay lập tức 100% các công cụ AI đọc hiểu được file repo thông qua AGENTS.md.
  • Adapter pattern tối giản giúp dễ dàng viết thêm adapter mới trong vòng dưới 1 tuần.
  • Nền tảng kiểm thử khá vững chắc (60 tests pass) đảm bảo không xảy ra regression lỗi khi refactor code core/CLI.
  • Phạm vi (scope) trong SA design ban đầu quá rộng so với nguồn lực MVP. Cần giữ MVP thật tinh gọn để sớm kiểm chứng giá trị cốt lõi trước khi làm registry hay marketplace.
  • Cơ chế validation đã giải quyết triệt để rủi ro silent no-op của các trường chưa hỗ trợ (báo lỗi dừng chương trình thay vì bỏ qua âm thầm).
  • Tệp template và policy hiện đang là liên kết tham chiếu tĩnh cho AI agent chứ chưa inline trực tiếp nội dung vào prompt.
  1. Adapter Contract: Đưa ra tệp quy ước ADAPTER_CONTRACT.md định hình rõ interface, tính deterministic và yêu cầu viết snapshot test cho adapter.
  2. Reference Validation: Bổ sung kiểm tra liên kết chéo trong project-validation.ts để tránh link gãy đến template/policy.
  3. Generated Output Strategy: Quyết định commit các tệp generated trực tiếp lên Git để sử dụng ngay kèm theo drift detection.
  4. MVP Exit Criteria: Xác lập tiêu chí đóng Phase 1 bao gồm validate chặt, build idempotent, adapter tests, CI xanh, README hoàn chỉnh.
